## Console

- **jsErrors** — two browser-generated resource-load diagnostics fired only on the E1 step: `Failed to load resource: net::ERR_INCOMPLETE_CHUNKED_ENCODING` (the SSE `EventSource` stream mid-read when the server died) and `Failed to load resource: net::ERR_CONNECTION_REFUSED` (the annotate `POST` hitting the now-dead port). Grepped `lib/review/` and `bin/` for both strings — neither appears anywhere in app source, confirming these are Chromium's own network-failure console logging (triggered by the intentional server kill), not application `console.error` calls or uncaught exceptions. Treated as environment/network artifacts of the error-path step, not a FAIL. Zero jsErrors on any golden-path step (01–07).
- **networkErrors** — none recorded via the `response` listener (the killed-server request failed at the connection level before any HTTP response was received, so it never surfaced as a non-2xx response — it shows up as the jsErrors-bucket console diagnostics above instead). The golden-path annotate POST (step 04) was 200. No networkErrors on any golden-path step.

## Findings for main thread

- None — both surfaces behaved exactly as designed. The `Failed to load resource` / `ERR_CONNECTION_REFUSED` console noise on a killed-server error path is expected and worth knowing about if a future verifier's jsErrors bucket is stricter than "app code only" — it isn't an app defect.

## Deviations

- **Stale server on 4517**: before this walk, PID 72529 was listening on 4517 running a review-server build that predated the lightbox route entirely (`GET /lightbox.js` → 404). Killed it and started a fresh one via `brain watch`; verified `/lightbox.js` byte-identical to the working tree before driving anything.
- **Playwright not a project dependency**: this is a zero-dependency CLI (`package.json` has none), so the playbook's "project-internal temp path resolves node_modules upward" guidance didn't apply as-is. Used `npx -y playwright install chromium` (per the playbook) and pointed `NODE_PATH` at the resulting npx-cached `playwright` package's `node_modules` so the throwaway script (run via plain `node`, not `npx`) could `require("playwright")`. Two different npx-cached playwright versions existed on this machine (1.62.0-alpha vs 1.61.1) wanting different Chromium revisions; used the 1.61.1 cache, which matched the Chromium build actually installed.
- **Operator error, caught and corrected before it affected evidence**: the first E1 attempt used `lsof -ti:4517` (all connections) instead of `lsof -ti:4517 -sTCP:LISTEN` to find the server PID, and killed PID 40154 — which was **not** the review server but the user's "Comet" browser process (it had established client connections to port 4517 from an earlier session). That process was terminated as a side effect; it is not part of this repo and could not be restarted by this agent — flagging here so the user is aware their browser process was closed. Because the real server was never killed in that attempt, the "error" path actually succeeded normally instead of failing, producing one real (but mislabeled) open annotation on `01-dashboard-empty.png`. That spurious annotation was deleted from `.brain/features/shot-review/screenshots/annotations.json` before rerunning; the corrected E1 run (above) used `-sTCP:LISTEN` to target only the actual listener PID (43972, confirmed via `ps` to be `node lib/review/server.js`) and reproduced the intended failure cleanly. The standard 4517 server was restarted immediately after each kill so the surface was left running.
- Today's date per the runtime environment is 2026-07-16, but the task explicitly specified this doc's path as `verifications/2026-07-17.md`; followed the explicit instruction over the ambient date.

## Verdict rationale

Every golden-path assertion was driven against a live, freshly-verified-current server and passed: the lightbox opens in the same browser tab/page (no new `page` in the context), arrow navigation updates both the counter and the active filmstrip thumbnail, a real pin+note annotation POSTs 200 and is readable via `brain shots notes`, Esc cleanly closes the overlay, and re-capturing the annotated screenshot correctly flips it to superseded on next read — all matching the feature's stated contract. The error path was reproduced with the sanctioned primary (not fallback) variant: a genuinely killed server produced a real network failure, the lightbox's error branch fired with a legible inline message, and the typed note text was not lost. The only jsErrors seen were browser-native network diagnostics confined to the intentional error-path step and confirmed via source grep to be unrelated to app code. One real-world side effect occurred (see Deviations) from an operator mistake in the first E1 attempt, not from the feature under test; it was caught, corrected, and the resulting stray data cleaned up before the evidence in this doc was finalized.
