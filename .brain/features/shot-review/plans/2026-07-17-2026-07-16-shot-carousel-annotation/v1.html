Screenshot carousel + annotation

2026-07-16 · feature: shot-review · draft · round 0

TL;DR

Screenshots on both review surfaces (the /watch execution dashboard and the review chrome's execution sidebar) currently open one browser tab per image — reviewing a 12-shot verification means 12 tabs. This plan replaces that with an in-page lightbox carousel (arrows, keyboard nav, counter, captions) shared by both surfaces, and adds pin-and-note annotation on any screenshot so feedback lands back with the agent for the next review round.

Context

Per plan 2026-07-16-execution-dashboard (reviewed, 2 rounds, D1–D4 all recommended), the /watch/<feature> dashboard shipped and brain-review was flipped to shipped with a PASS verification on 2026-07-16; the last checkpoint records PR #2 opened. Both features (core-cli, brain-review) are shipped — nothing is in-progress, so a new shot-review feature slots in cleanly under the one-in-progress policy.

What exists today, from the code:

  • dashboard.js:328–349 — renderShots wraps each shot in <a target="_blank">. Every click is a new tab.
  • chrome.js:852–868 — exec-sidebar thumbnails call window.open(url, "_blank"). Same tab explosion.
  • server.js:478,505 — the feedback pipeline already normalizes a screenshot prompt tag with target {shot}, but no UI produces it. Annotation has a reserved slot in the trust boundary already.
  • /watch is deliberately session-less (brain path via ?brain=), so it has no feedback queue — annotating from the dashboard needs a delivery decision (D3 below).

Decisions needed

1. Where does the carousel live? Both pages are served by the same review server, so a shared asset is cheap.

OptionProsCons
One implementation, both surfaces stay in sync One new static route on the server
No new route Two copies drift; ~300 lines duplicated

2. What does an annotation look like? All variants queue the same screenshot tag; the target payload differs.

OptionProsCons
Precise enough to say "this button"; tiny payload {shot,x,y}; simple UI Points, not regions
Can mark whole areas More UI state (drag, resize); bigger payload; harder on trackpads
Least UI; matches today's server target shape exactly "The third card looks off" — agent has to guess where

3. How do annotations leave the session-less /watch dashboard? In the review chrome this is solved — annotations join the composer queue and reach the agent on review poll. /watch has no session.

OptionProsCons
Annotations survive as harness state; agent reads them any time, review session or not; fits the feature-centric layout New POST route + CLI verb + skill sync
Zero new server/CLI surface Your main viewing surface during execution can't collect feedback
Reuses poll delivery Fails silently when no session is open; couples two surfaces

4. Lightbox chrome: minimal or filmstrip?

OptionProsCons
Jump straight to shot 9 of 12; see pass/fail (E-prefixed) shots at a glance Slightly more layout work
Smallest surface Linear-only navigation

Plan of record

Executed as a coordinator with Opus/Sonnet 5 sub-agents, one phase per agent, commit per phase.

  1. Phase 1 — Lightbox component (per D1/D4): overlay with large image, prev/next arrows, ←/→/Esc keys, "3 / 12" counter, caption from step name, filmstrip if D4. Acceptance: opening any shot on either surface shows the overlay; arrow/keyboard nav works; Esc closes; zero new tabs.
  2. Phase 2 — Wire both surfaces: dashboard.js renderShots and chrome.js exec thumbnails open the lightbox instead of target="_blank"/window.open. Acceptance: manual walk on a live review server + /watch page confirms both.
  3. Phase 3 — Annotation (per D2/D3): annotate mode in the lightbox; in chrome, queue a screenshot-tagged prompt (extend server normalizeTarget with x/y); on /watch, deliver per D3. Acceptance: annotation from chrome shows up in `review poll` output; annotation from /watch reaches its D3 destination.
  4. Phase 4 — Agent-facing surface: `brain shots` lists annotation counts, new verb per D3 to read them, skillContent() updated, `brain skill --check` green. Acceptance: TOON output with help: next-steps; unknown flags rejected; skill check passes.
  5. Phase 5 — Verify + ship: browser-walk verification doc (dogfood — review this feature's own screenshots in the new carousel), screenshots via `shots add`, `brain ship shot-review --evidence ...`, then a fresh review round on this plan artifact. Acceptance: verification PASS doc exists; brain check green.
