annotation-watch — post-ship feedback runner

2026-07-17 · feature: annotation-watch · draft · round 0

TL;DR

Pins dropped on the /watch dashboard persist correctly but nothing wakes the agent — feedback strands in annotations.json until someone happens to run shots notes. This plan adds brain watch poll <feature>, a blocking runner built exactly like review poll — it long-polls the review server, and the reviewer's Send to Claude click resolves it with the sent batch. The agent starts it when a feature reaches the shipped/PR state; while connected the dashboard shows an "agent watching" badge and the post-send toast says feedback was picked up live — the same feedback-round experience the plan phase already has.

Context

shot-review shipped 2026-07-17 (PR #4) with pin+note annotation on both surfaces. Tracing the loop end-to-end found an asymmetry:

  • Chrome (plan phase): annotation → composer queue → Send → review poll returns it. Closed loop; the CTA toast points at Send.
  • /watch (execution/shipped phase): annotation → POST → annotations.json → silence. No proactive surface (home, progress, watchContext, playbooks) mentions open annotations; only brain shots shows a count if the agent happens to run it.

The dashboard's pipeline already renders a terminal shipped → PR opened state (execution-dashboard plan, D1–D4, 2026-07-16) — exactly the moment a human starts reviewing screenshots and pinning feedback, and exactly when the agent has stopped listening today.

Decisions needed

1. How does the runner get woken + how does the dashboard know it's there? DECIDED (user, in chat): server long-poll, exactly like review poll. The runner connects to the same review server the dashboard runs on; presence is instant (a connected poller = agent watching); clicking Send to Claude resolves the waiting poll with the sent batch — the plan phase's composer→poll contract, applied to features. The earlier heartbeat-file idea predated the Send step and is dropped: pins can only be created through the running server anyway, so "works without a server" bought nothing.

Plan of record

Coordinator + Opus/Sonnet 5 sub-agents, one phase per agent, commit per phase, on a new branch feat/annotation-watch stacked on feat/shot-review-carousel (PR #4).

  1. Phase 1 — Runner CLI + server (per D1/D2): `brain watch poll <feature>` — blocking, mirrors `review poll`: long-polls GET /watch/<feature>/poll on the review server; on connect first drains any sent-but-undelivered annotations (sentAt set, deliveredAt null), else waits until Send resolves it; each delivered record stamped deliveredAt; --timeout (default 600s) exits 0 with status: timeout + re-run help; no server → opError + `brain watch` help. TOON output ends with the CTA help loop: apply fix → shots add (supersedes) → poll again. skillContent sync. Acceptance: Send resolves a waiting poll <2s; drain-on-connect works; timeout/unknown-feature/no-server paths verified; re-run never double-delivers; skill --check green.
  2. Badge toggles via the existing context SSE without reload. Acceptance: badge appears on poll connect and clears on disconnect (SSE-fast, no TTL lag); toast variant correct in both states.
  3. Phase 3 — Shipped-section integration (per D2/D4): `brain ship` and `brain pr` help lines nudge `shots poll`; execute playbook gains the post-ship watch step; home + progress print an open-annotations line when any exist (D4). skill sync. Acceptance: help lines present; home/progress line appears only with open annotations; skill --check + brain check green.
  4. Phase 4 — Verify + ship: Playwright walk — runner attached, badge visible, pin a note, poll returns it live, re-capture supersedes, badge clears after runner exit; E1 error path; verification doc; brain ship annotation-watch --evidence.
